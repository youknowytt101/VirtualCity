### H-001｜Blast SOP `negate` 参数方向反转

**现象**
建筑和道路数量对调：建筑显示 308 个，道路显示 773 个（实际应相反）。

**根因**
Blast SOP 的 `negate` 参数默认值为 0：
- `negate = 0`：**删除**指定 group，保留其余（默认，反直觉）
- `negate = 1`：**保留**指定 group，删除其余（我们需要的）

**修复**
```python
blast_node.parm('negate').set(1)
```

**规则**
> 在 Houdini 中用 Blast SOP 提取某个 group 时，**必须显式设置 `negate=1`**，否则默认删除该 group。

### H-002｜OSM 坐标 `-z` 翻转导致 signed_area 方向判断反转

**现象**
将 signed_area > 0 时翻转顶点顺序后，所有建筑全部朝下拉伸（入地）。

**根因**
OSM 坐标转 Houdini 本地坐标时，纬度映射到 Z 轴并**取反**：
```python
p.setPosition(hou.Vector3(x, 0, -z))  # z 被取反！
```
这导致 2D signed_area 的 CCW/CW 方向在 Houdini 中翻转：
- local (x, z) 空间中 SA > 0 = CCW
- 但存入 Houdini (x, -z) 后，CCW 变 CW → 法线朝 **-Y（向下）**
- 因此需要在 SA > 0 时翻转（而非 SA < 0）

**正确代码**
```python
sa = signed_area_xz(local_pts)  # 用原始 (x, z) 计算
if sa > 0:                        # >0 表示 Houdini 中法线朝下，需要翻转
    local_pts = list(reversed(local_pts))
# 之后再用 Vector3(x, 0, -z) 存储
```

**规则**
> OSM → Houdini 坐标转换时，如果 z 取反，则 signed_area 的正负判断方向与直觉相反，**SA > 0 时需要翻转顶点顺序**。

### H-003｜OSM 建筑多边形需要 Fuse + Divide 才能正确拉伸

**现象**
直接对 OSM 建筑轮廓做 PolyExtrude，会出现黑面、三角形缺口、面片穿插等问题。

**根因**
OSM 建筑轮廓存在以下问题：
1. **重复顶点**：第一个和最后一个 node 通常相同（OSM 闭合方式）
2. **凹多边形**：许多建筑形状为 L/U/T 形，不可直接正确拉伸
3. **微小误差**：相邻顶点距离 < 0.01m，导致退化三角形

**修复**
在 `extrude` 前插入两个节点：
```python
# Fuse：合并距离 < 0.01m 的重复点
fuse = geo_node.createNode('fuse', 'fuse_bld')
fuse.parm('snaptype').set(1)

# Divide：三角化，处理凹多边形
divide = geo_node.createNode('divide', 'divide_bld')
divide.parm('convex').set(1)
divide.parm('numsides').set(3)
divide.parm('noslivers').set(1)
```

**规则**
> OSM 建筑轮廓在拉伸前**必须经过 Fuse + Divide 三角化**。

### H-004｜PolyWire 生成管道不适合道路显示

**现象**
使用 `polywire` SOP 后道路变成圆柱管道，面数暴增至 345,000+，视口卡顿且不符合白盒效果。

**替代方案**
使用 Python SOP 手动生成扁平道路条带（沿切线方向做垂直偏移）：
```python
tang = (positions[i+1] - positions[i-1]).normalized()
perp = UP.cross(tang).normalized()
left_pt  = pos + perp * half_width
right_pt = pos - perp * half_width
```

**规则**
> 道路面化不要用 `polywire`（生成圆管）。MVP 阶段用 Python SOP 生成扁平条带，或用 `polyexpand2d`（注意：polyexpand2d 对开放曲线支持有限）。

### O-001｜Overpass 查询 bbox 纬度偏高导致空数据集

**现象**
Overpass 查询正常完成但返回 0 个元素。

**根因**
初始 bbox 估算偏高约 1.5 km（纬度偏大），查询范围偏移到海面。

**修复**
将 south/north 纬度下调约 0.015°（≈1.67 km），重新验证直到有数据返回。

**规则**
> 首次 Overpass 查询后，检查返回元素数量。若为 0，优先检查 bbox 是否偏移，不要假设查询语法有误。

## H-006：GeoJSON feature geometry 为 None 导致 Python SOP 崩溃

**现象**：`osm_import` Python SOP 报 `TypeError: 'NoneType' object is not subscriptable`，位置在读取 `geom['coordinates']` 处。

**根因**：Google Open Buildings GeoJSON 中少数 feature 的 `geometry` 字段为 `null`。

**修复**：在访问 `geom['coordinates']` 前加 `if geom is None: continue`。

**规则**
> 读取 GeoJSON geometry 前必须判空，`if geom is None: continue`。

## D-001：EE `getDownloadURL(selectors=...)` 导致 GeoJSON geometry 全部为 null

**现象**：下载的 GeoJSON 有 2046 个 feature，height 全部正常，但 `geometry` 字段全是 `null`，Houdini 建筑全部不显示。

**根因**：`FeatureCollection.getDownloadURL(filetype="GeoJSON", selectors=["area","height","presence"])` 中 `selectors` 参数在某些 EE 版本下会剥离 geometry。同时 `f.transform(proj=...)` 会把坐标系转为投影坐标，GeoJSON 序列化时丢失。

**修复**：
1. 去掉 `selectors` 参数，改为用 `ee.Feature(f.geometry(), {...})` 手动重建 feature 保留 geometry。
2. 去掉 `.transform()` 调用，保持 WGS84 坐标。
3. 验证时同时检查 `geometry` 是否为 null，发现全 null 立即报错。

**规则**
> EE 下载 GeoJSON 时不要用 `selectors`；用 `ee.Feature(geom, props)` 手动构建保证 geometry 存在。

## H-007：`triangulate2d` + `restorepos=1` 高度值异常，`heightfield_file` 无法读 int16 GeoTIFF

**现象**：用 `triangulate2d` 做 DEM 地形，`restorepos=0` 时 Y 值正确（-8~29m），`restorepos=1` 后 Y 值变为 0~99m（错误）；改用 `heightfield_file` 读 int16 GeoTIFF，Y 输出全为 0。

**根因**：
- `triangulate2d` 的 `restorepos` 在有新增点（refinement）时无法正确还原所有点的原始 Y 坐标。
- `heightfield_file` SOP 无法正确解析 int16 类型且无 proj 信息的小尺寸 GeoTIFF。

**修复**：放弃 `triangulate2d` 和 `heightfield_file`，改用 Python SOP 直接从 DEM CSV 构建规则格网（105×105），手动创建四边形面片，Y 值直接赋高程，稳定可靠。

**规则**
> DEM 地形在 Houdini 中优先用 Python SOP 从 CSV 构建规则格网，不依赖 triangulate2d 或 heightfield_file。

## H-008：Houdini FBX 导出单位不转换，UE5 资产缩小 100 倍

**现象**：FBX 导入 UE5 后资产尺寸是 Houdini 中的 1/100（建筑 95m → 0.95m，城市 1800m → 18m）。

**根因**：Houdini 坐标单位是**米**，FBX 标准默认单位是**厘米**。`filmboxfbx` ROP 的 `convertunits=0`（不转换）导出时原样写入米值，UE5 读取 FBX 时把这些值当厘米解析，造成 1/100 缩放。

**错误尝试**：在 UE5 `FbxImportUI` 里设 `import_uniform_scale=100` 反而让资产更小（该参数含义与预期相反，或被应用了两次）。

**正确修复**：`filmboxfbx` ROP 设置 `convertunits=1`，Houdini 在导出时自动把米制坐标 ×100 写入 FBX，UE5 导入即为真实厘米单位，无需额外缩放。

**规则**
> Houdini → UE5 FBX 导出必须设 `convertunits=1`；UE5 导入侧不加任何 `import_uniform_scale`。
> 验证方法：导入后用 Remote Control 查 `ExtendedBounds.BoxExtent.X`，应约等于 Houdini 中 X 范围（米）× 100。

## U-002：UE5 编辑器持续闪屏 / 画面撕裂

**现象**：UE5 Editor 视口持续闪烁或画面撕裂，重启后仍然存在。

**根因**：`Saved/Config/WindowsEditor/GameUserSettings.ini` 中 `bUseVSync=False` + `FrameRateLimit=0`，GPU 以极高帧率渲染（数千 FPS），超出显示器刷新率，导致持续撕裂。此文件由 UE5 自动生成，不受 `DefaultEngine.ini` 初始值约束。

**注意**：在 `DefaultEngine.ini` 中加 `[/Script/RemoteControl.RemoteControlSettings]` + `+AllowedClasses=...` 是无效节，虽然不是闪屏原因，但会产生 UE5 启动警告，应删除。

**修复**：在 `DefaultEngine.ini` 的 `[/Script/Engine.RendererSettings]` 节加 `r.VSync=1`，并在 `[/Script/Engine.Engine]` 节加 `MaxFPS=60`，强制锁帧同步显示器。

**规则**
> UE5 项目创建后立即在 `DefaultEngine.ini` 加 `r.VSync=1` 和 `MaxFPS=60`，避免无限帧率撕裂。

### H-009｜道路条带侧边顶点埋入地形

**现象**
坡面区域道路左右边缘陷入地形，平视角明显可见道路一侧或两侧消失在地表以下。

**根因**
`road_strips` Python SOP 生成侧边点时，只继承中心线点的 Y 值（`pos + perp * half_width` 中 Y 不变）。坡面上，侧边偏移后所在 XZ 位置的地形高度与中心线 Y 不同，导致侧边点与地形高度不匹配。

**修复**
在 `road_strips` 后新增 `snap_road_strips` attribwrangle（`class=2`），对所有条带顶点做二次地形投影：
```vex
int hit_prim;
vector uvw;
xyzdist(1, @P, hit_prim, uvw);
vector tp = primuv(1, "P", hit_prim, uvw);
@P.y = tp.y + 0.15;  // +0.15m 防 z-fighting
```
input0 = road_strips，input1 = dem_subdivide。

**规则**
> 任何从中心线扩宽生成的道路网格，必须在最终输出前对**全部顶点**（含侧边）再做一次 `xyzdist` 地形投影。`road_clipped` 的源节点应为 `snap_road_strips`，不是 `road_strips`。

### H-010｜地形网格过稀，坡度呈阶梯状

**现象**
Houdini/UE5 中地形出现明显棱角和阶梯，建筑/道路贴地不平滑，布线可见（2km² 仅 4488 个四边形）。

**根因**
NASADEM/SRTM 30m 精度，每个 DEM 格子覆盖 30m×30m，远粗于建筑（~10m）和道路宽度（~7m）。CSV 导入后直接构建格网，未做任何加密。

**修复**
在 `dem_terrain` 后新增 `dem_subdivide`（Subdivide SOP）：
```python
dem_subdiv.parm('algorithm').set(4)   # OpenSubdiv Bilinear（线性插值，不改变原始高程）
dem_subdiv.parm('iterations').set(2)  # 2 轮 = ×16 面片（4488 → 71808，等效 ~7.5m）
```
同时将 `snap_bld_to_terrain`、`snap_roads_to_terrain1`、`merge_all` 的地形输入全部改为 `dem_subdivide`。

**注意**
- `iterations=2` = 两轮细分 = ×4×4 = **×16**，不是 ×2
- `algorithm=4`（Bilinear）仅线性插值，保持原始高程不变；不要用 Catmull-Clark（会使地形过度平滑变形）
- 加密只改善视觉平滑度，不增加真实高程精度（精度仍受限于 30m 数据源）

**规则**
> DEM 导入后必须接 `dem_subdivide`（Bilinear iterations=2），所有下游吸附节点 input1 指向 `dem_subdivide`，不直接使用 `dem_terrain`。

### H-011｜坡面建筑底面被地形埋没（质心单点吸附）

**现象**
丘陵/坡面区域部分建筑底面陷入地形，从侧面看建筑下半部分消失在地表以下。平坦区域建筑正常。

**根因**
`snap_bld_to_terrain` 原始 VEX 用 `@P`（Primitive 质心）查一次地形高度，将建筑所有顶点统一设为质心处的 Y 值。坡面上：
- 质心 Y = 20m（坡中）
- 上坡侧顶点实际地形 = 30m → 底面 Y=20 < 地形 30 → 上坡侧被埋 10m

**修复**
改为对每栋楼逐顶点查询地形高度，取 **MAX** 作为底面统一 Y：
```vex
float max_y = -1e9;
int verts[] = primvertices(0, @primnum);
foreach(int v; verts) {
    int pt = vertexpoint(0, v);
    vector p = point(0, "P", pt);
    int hit_prim; vector uvw;
    xyzdist(1, p, hit_prim, uvw);
    float ty = primuv(1, "P", hit_prim, uvw).y;
    if (ty > max_y) max_y = ty;
}
// 所有顶点统一设为 max_y
```
效果：建筑底面 Y_min 从 0.0m 升至 4.6m，0 个顶点被埋。

**权衡**
使用 MAX 会让建筑在下坡侧轻微悬浮（建筑漂浮而非被埋），这在视觉上比被埋更可接受。如需更精确，可对每栋楼底面做 terrain conforming（逐顶点独立设 Y），但会使底面不水平。

**规则**
> `snap_bld_to_terrain` 必须用逐顶点 MAX 地形高度，禁止用质心单点吸附。坡面建筑优先保证"不被埋"而非"完全贴合"。

## A-001：自动化脚本硬编码盘符导致跨机器失效

**现象**
家里机器使用 `F:/VirtualCity`，公司机器使用 `d:/VirtualCity`。Git 同步后，脚本在另一台机器上仍尝试访问旧盘符路径，导致数据文件、Hip、UE5 trigger 找不到。

**根因**
多个脚本直接写死：
```python
ROOT = Path(r"F:/VirtualCity")
CFG_FILE = r"F:/VirtualCity/配置/active_area.json"
TRIGGER_FILE = r"F:/VirtualCity/.ue5_trigger"
```

**修复**
新增 `自动化插件/vc_paths.py`，所有脚本通过脚本位置推导项目根目录：
```python
ROOT = Path(__file__).resolve().parents[1]
```
并统一使用：
```python
from vc_paths import ROOT, DATA_ROOT, HIP, EXPORT, load_active_area
```
`active_area.json` 中保存相对路径，运行时由 `resolve_project_path()` 转成当前机器绝对路径。

**规则**
> 自动化脚本和配置文件禁止硬编码 `F:/VirtualCity` / `D:/VirtualCity` 等机器专属盘符。所有路径必须通过 `vc_paths.py` 推导；配置文件优先保存相对路径。

## A-002：Windows GBK 控制台输出 emoji 导致 `UnicodeEncodeError`

**现象**
`validate_data.py` 数据检查已通过，但最后输出 `✅` / `❌` 时在 Windows PowerShell GBK 环境中崩溃：
```text
UnicodeEncodeError: 'gbk' codec can't encode character
```

**根因**
Windows 默认控制台编码可能不是 UTF-8，emoji 或部分特殊符号无法编码。

**修复**
关键状态输出改用 ASCII 标记：
```python
OK = "[OK]"
FAIL = "[FAIL]"
WARN = "[WARN]"
```

**规则**
> 自动化脚本的关键 CLI 状态输出优先使用 `[OK] / [FAIL] / [WARN]`，不要依赖 emoji。中文说明可以保留，但关键流程不能因字符编码失败而中断。
