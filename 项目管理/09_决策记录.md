## ADR-001｜主路线采用 OSM + DEM + Houdini + UE5

- 状态：Accepted
- 日期：2026-05-21

### 背景

项目目标是生成可编辑、可控制、可扩展的虚拟城市，而不是只做真实地图浏览。

### 决策

第一阶段主路线采用：

```text
OSM / DEM
    ↓
Houdini 程序化生成
    ↓
UE5 Houdini Engine / Bake
```

### 原因

- OSM 免费且易获取。
- Houdini 适合道路、建筑、地形的程序化处理。
- UE5 适合最终展示、Bake、Nanite、Lumen、PCG。

### 影响

- Mapbox 不作为主线。
- Cesium 只作为参考或数字孪生方向。

## ADR-002｜第一阶段只做 200m～500m MVP 小区块

- 状态：Accepted
- 日期：2026-05-21

### 背景

完整城市或 City Sample 级管线成本太高，会放大坐标、性能、Cook 时间和资产管理问题。

### 决策

第一阶段只做 `200m × 200m` 到 `500m × 500m` 的测试区域。

### 原因

- 更容易验证 OSM → Houdini → UE5 主链路。
- 更容易定位坐标和比例问题。
- 更适合快速迭代。

### 影响

- 暂不做整城级数据。
- 暂不启用复杂 World Partition 和 PDG。

## ADR-003｜MVP 阶段不优先开发自研插件

- 状态：Accepted
- 日期：2026-05-21

### 背景

当前还没有跑通第一条完整人工流程，过早开发工具会增加复杂度。

### 决策

MVP 阶段优先手动跑通流程，不开发大型自研插件。

### 原因

- 手动流程更快验证真实问题。
- 只有重复 3 次以上的步骤才值得脚本化。
- 避免把时间花在未验证的自动化上。

### 影响

- `Scripts/` 仅作为规划和预留位。
- 后续脚本开发必须遵守 `Scripts/工具开发规范.md`。

## ADR-004｜AI 接手入口固定为项目管理目录

- 状态：Accepted
- 日期：2026-05-21

### 背景

用户希望每次 AI 打开项目后都能快速找到资料，并根据项目持续自动迭代。

### 决策

固定入口为：

```text
项目管理/00_AI接手指南.md
项目管理/02_当前状态与下一步.md
项目管理/01_资料地图.md
项目管理/project_manifest.json
```

### 原因

- Markdown 适合人类和 AI 阅读。
- JSON 适合未来脚本或 AI 自动解析。
- 状态、资料、日志、决策分离，减少混乱。

### 影响

- 每次阶段性任务后必须更新状态和日志。
- 新增关键资料时必须更新资料地图和 manifest。

## ADR-005｜Houdini 内部采用米制，UE5 导入统一缩放

- 状态：Accepted
- 日期：2026-05-21

### 背景

Houdini 和 UE5 单位不同，容易导致导入后比例错误。

### 决策

采用：

```text
Houdini：1 unit = 1 meter
UE5：1 Unreal unit = 1 centimeter
Houdini → UE5：统一乘以 100
```

### 原因

- GIS / OSM / DEM 更适合米制处理。
- UE5 内部默认厘米。
- 集中缩放比多节点分散缩放更容易维护。

### 影响

- 缩放应集中在 HDA 输出端或 UE 导入配置中。
- 不建议在多个 Houdini 节点中分散处理单位换算。

## ADR-006｜配置文件用于自动化，Markdown 用于解释和记录

- 状态：Accepted
- 日期：2026-05-21

### 背景

项目既需要人类和 AI 理解上下文，也需要未来脚本读取参数。

### 决策

- Markdown 负责解释流程、记录决策和状态。
- JSON 配置负责区域、管线和 QA 的机器可读参数。

### 原因

- Markdown 可读性强。
- JSON 更适合自动化和校验。
- 两者分工明确，避免把所有信息混在一个文档里。

### 影响

- 新区域应同时创建区域记录 Markdown 和区域配置 JSON。
- 管线参数应优先进入 `Config/` 模板或实例配置。

## ADR-007｜区域接入由 AI 负责整理，用户只提供框选结果

- 状态：Accepted
- 日期：2026-05-21

### 背景

用户反馈，每次框选区域后需要手动整理多个文档、配置和查询模板，操作重复且容易遗漏。

### 决策

明确分工：

- **用户负责**：在地图工具中框选区域，并向 AI 提供 bbox / GeoJSON / 截图 + 地点名称中的任意一种。
- **AI 负责**：生成 `area_id`，创建区域记录 Markdown、区域配置 JSON、Overpass 查询模板，更新当前状态、任务看板、project manifest 和迭代日志。

### 原因

- 区域接入工作全部可自动化，无需用户手动编辑。
- 减少用户认知负担，让用户专注于框选位置和验收结果。
- AI 整理结果统一格式，减少文档不一致风险。

### 约束

- AI 不得伪造 bbox。如果输入不含精确经纬度，状态设为 `bbox_pending`。
- AI 不得覆盖已存在的区域记录，除非用户明确要求更新。

### 影响

- 本偏好已写入：
  - `05_自动迭代协议.md`（第 3.1 节）
  - `10_AI启动自检清单.md`（第 7 节）
  - `.windsurf/workflows/area-setup.md`（第 4、5、6 节）

## ADR-008｜“重新测试”必须从 area_picker 网页入口开始

- 状态：Accepted
- 日期：2026-05-28

### 背景

快速迭代中曾把直接运行 `set_area.py` 或 `_recook_new_area.py` 误当成完整流程测试，导致跳过网页框选、数据获取或 QA 环节。

### 决策

用户说“重新测试 / 从头测试 / 全流程测试 / 测试自动化管线”时，默认必须从：

```bash
cd Scripts
uv run python area_picker.py
```

开始，流程结束条件为同一区域：

- `Config/houdini_build_status.json` 中 `status=completed`
- `Reports/model_qa/latest.json` 中 `status=pass`

### 原因

- `area_picker.py` 是用户级入口。
- `set_area.py` 只是底层 bbox 命令入口。
- `_recook_new_area.py` 只是 Houdini 当前区域重构入口。
- 完整测试必须覆盖数据获取、清洗、Houdini build 和 Model QA。

### 影响

- AI 不得把直接 recook 称为完整测试。
- 只有用户明确要求“跳过网页 / 当前区域重构 / 直接用这个 bbox”时，才运行底层入口。

## ADR-009｜Houdini Model QA 作为快速实验阶段的自动护栏

- 状态：Accepted
- 日期：2026-05-28

### 背景

道路异常大面片、坡地道路埋地、建筑裙边颜色/法线/对齐、建筑倒角短边等问题，仅靠人工视口容易漏检，也难以在快速迭代中稳定回归。

### 决策

在 `_recook_new_area.py` 构建完成后自动运行：

```bash
uv run python Scripts/houdini_model_qa.py --mode quick
```

Model QA quick 至少检查道路、地形、建筑、裙边、倒角和贴地状态。QA 不通过时，Houdini build status 不得标记为 completed。

### 原因

- 自动捕捉明显几何错误。
- 给每次区域测试留下 JSON 报告。
- 降低快速迭代时的回归风险。

### 影响

- `Reports/model_qa/latest.json` 成为当前流程的重要状态文件。
- QA 统计必须在 Houdini 进程内部完成，避免 RPYC 大几何遍历超时。
- Model QA 不替代人工审核，只作为自动护栏。
