# 07 补充优化建议：2026 版地图 API + Houdini + UE5 城市管线

> 基于当前 `01~06` 文档复盘，并结合 2025~2026 年主流 Houdini / UE5 / Cesium / GIS 城市制作流程整理。  
> 结论：现有文档主线正确，建议补强“数据治理、坐标转换、分块烘焙、PCG/World Partition、自动化流水线、国产地图合规”六个环节。

---

## 1. 总体评价

当前文档已经覆盖了最核心的行业路线：

```text
OSM / DEM / Cesium 数据获取
    ↓
Houdini 清洗、投影、道路/地块/建筑生成
    ↓
HDA / 点云 / Alembic / Landscape 输出
    ↓
UE5 Houdini Engine / PCG / World Partition / Nanite / Lumen 整合
```

这个方向是正确的。尤其是 `06_路线一详细拆解_Houdini+UE5城市管线.md` 中围绕 City Sample / Matrix Awakens 的拆解，已经接近当前 AAA 级虚拟城市制作的参考答案。

但如果要从“调研文档”提升到“可实际落地的生产管线规范”，建议补充以下内容：

- 数据进入 Houdini 前的 GIS 预处理规范。
- 大城市分块、版本管理、增量更新策略。
- Houdini Engine for Unreal 最新 PCG 版本差异。
- UE5 World Partition / Data Layers / HLOD 的生产级约束。
- 国内地图 API 的坐标系、合规、偏移处理。
- 自动化批处理脚本和质量检查清单。

## 2. 当前市场最优解流程

### 2.1 游戏 / 影视 / XR 城市

推荐主流程：

```text
数据源
  ├─ OSM / 政府 GIS / 自绘规划线
  ├─ DEM / LiDAR / 航测地形
  └─ 可选 Cesium / Google 3D Tiles 作为视觉参考

预处理
  ├─ QGIS / Python / GDAL 统一投影
  ├─ WGS84 / GCJ-02 / CGCS2000 坐标转换
  ├─ 道路、建筑、水系、绿地分类清洗
  └─ 按 Tile / Block / District 分块

Houdini
  ├─ OSM / GIS 导入
  ├─ 路网拓扑修复
  ├─ 地块划分
  ├─ 建筑体量生成
  ├─ 模块化立面 / Shape Grammar
  ├─ 点云属性输出
  └─ PDG 分布式或多核批处理

UE5
  ├─ Houdini Engine / HDA 迭代
  ├─ PCG 生成植被、垃圾、屋顶设备、小道具
  ├─ Nanite + HISM / ISM 实例化
  ├─ World Partition + Data Layers
  ├─ HLOD / Runtime Virtual Texture
  ├─ Mass Traffic / ZoneGraph
  └─ Bake 成稳定关卡内容
```

这是目前最稳妥的“可编辑、可美术控制、可游戏化”的方案。

### 2.2 数字孪生 / 城市规划 / 政务可视化

推荐主流程：

```text
权威 GIS / BIM / 倾斜摄影 / 3D Tiles
    ↓
Cesium for Unreal / ArcGIS Maps SDK / SuperMap / iTwin
    ↓
UE5 地理配准、数据图层、实时业务数据接入
    ↓
Houdini 只负责局部规划方案、程序化补全、视觉增强
```

这类项目不应完全照搬 City Sample。数字孪生优先级是：

1. 坐标准确。
2. 数据可追溯。
3. 图层可更新。
4. 合规可交付。
5. 最后才是美术质量。

## 3. 建议补充到现有文档的关键点

### 3.1 增加“GIS 预处理”章节

当前文档直接从 OSM / DEM 进入 Houdini，实际生产中中间通常需要 QGIS / GDAL / Python 清洗。

建议新增流程：

```text
原始 OSM / SHP / GeoJSON / DEM
    ↓
QGIS 检查坐标系
    ↓
统一投影到本地米制坐标系
    ↓
裁剪 bbox / 修复破碎几何 / 删除重复线
    ↓
导出 GeoJSON / SHP / OSM / GeoTIFF
    ↓
Houdini 导入
```

需要强调：

- Houdini 内尽量不要直接处理经纬度“大坐标”。
- 应先转为局部平面坐标，例如 UTM、本地 ENU 或项目自定义原点。
- UE5 大世界中也应设置世界原点和地理配准策略，避免浮点精度问题。

### 3.2 修正和统一单位描述

现有文档里同时出现了：

- “Houdini 1 unit = UE5 1 cm”。
- “Houdini 建议 1 unit = 1m，导入 UE5 需 ×100”。

建议统一为生产规范：

```text
Houdini 内部建模推荐：1 Houdini unit = 1 meter
UE5 内部单位固定：1 Unreal unit = 1 centimeter
Houdini → UE 输出时：统一乘以 100
UE → Houdini 输入时：统一乘以 0.01
```

并在 HDA 输出端集中处理 scale，不建议每个节点各自缩放。

### 3.3 增加“大城市分块策略”

当前文档讲到了 World Partition，但还缺少 Houdini 侧分块规则。

建议补充：

```text
城市范围
    ↓
按 250m / 500m / 1km Tile 切分
    ↓
每个 Tile 独立生成：建筑、道路、街具、植被点云
    ↓
跨 Tile 道路和水系单独维护为 Network Layer
    ↓
UE5 中对应 World Partition Cell / Data Layer
```

推荐分层：

| 层级 | 内容 | 更新频率 |
|------|------|----------|
| Terrain Tile | Landscape / 地形 | 低 |
| Road Network | 主路、辅路、高架 | 中 |
| Building Lots | 地块、建筑体量 | 中 |
| Facade Instances | 建筑模块实例 | 高 |
| Props / Decals | 街具、贴花、垃圾 | 高 |
| Runtime Agents | 车辆、行人、事件 | 运行时 |

### 3.4 增加“增量更新”策略

真实项目不会每次全城重算。建议记录每个 Tile 的输入 Hash。

```text
bbox / road source / lot rules / seed / HDA version
    ↓
生成 hash
    ↓
只有 hash 变化的 tile 才重新 cook
```

建议输出 manifest：

```json
{
  "tile_id": "city_012_008",
  "source_bbox": [0, 0, 500, 500],
  "hda_version": "citygen_1.4.2",
  "rule_seed": 23891,
  "outputs": [
    "buildings.pbc",
    "roads.abc",
    "props.pbc"
  ]
}
```

### 3.5 增加“Houdini Engine PCG 版”说明

Houdini Engine for Unreal 现在通常分为普通版和 PCG 支持版。建议在 `03_UE5整合方案.md` 中补充：

- 如果项目不用 UE PCG，使用普通 Houdini Engine 更稳定。
- 如果项目要让 HDA 作为 PCG Graph 节点参与生成，使用 PCG 版插件。
- PCG 版会引入额外依赖，团队要统一插件版本。
- 对大世界，推荐 Houdini 负责“结构性生成”，PCG 负责“环境填充”。

推荐职责划分：

| 系统 | 适合负责 | 不适合负责 |
|------|----------|------------|
| Houdini | 路网、地块、建筑体量、立面模块、桥梁 | 运行时高频变化 |
| UE PCG | 植被、垃圾、贴花、屋顶设备、局部道具 | 复杂路网拓扑求解 |
| Mass / ZoneGraph | 车辆、行人、动态仿真 | 静态城市建模 |

### 3.6 增加 HLOD / Nanite / ISM 的性能约束

文档中已提 Nanite + ISM，但建议补充真实限制：

- Nanite 适合静态高模建筑模块，不等于可以无限堆材质槽。
- ISM / HISM 可以显著降低 Draw Call，但实例过多时仍要分 Tile / 分 Layer 管理。
- 透明材质、玻璃、动态灯、Decal 会成为城市性能瓶颈。
- World Partition 大地图必须配合 HLOD，否则远景 Actor 数量仍会过高。
- 建筑窗灯建议使用材质参数、贴图 Mask 或实例随机，不建议每扇窗放真实灯光。

建议增加性能目标表：

| 场景级别 | 建议目标 |
|----------|----------|
| 编辑器预览 | 30 FPS 可操作 |
| 高端 PC 运行 | 60 FPS |
| 开放世界城市 | 稳定流送，无明显 hitch |
| VR / XR | 90 FPS，需更激进 LOD |

### 3.7 增加“中国地图数据专项流程”

现有文档提到 GCJ-02，但建议展开为独立章节。

国内项目建议流程：

```text
天地图 / 高德 / 百度 / 政府 GIS
    ↓
确认坐标系：WGS84 / GCJ-02 / BD-09 / CGCS2000
    ↓
合法授权与数据脱敏
    ↓
坐标转换到项目本地坐标
    ↓
Houdini 程序化生成
    ↓
UE5 地理配准或局部世界坐标
```

特别注意：

- 高德、百度瓦片和矢量数据通常不等同于 WGS84。
- 商业项目必须确认数据授权范围，包括是否允许离线化、二次加工、引擎内发布。
- 如果用于政府、园区、交通项目，优先使用甲方提供的 SHP / GeoJSON / CAD / BIM 数据，而不是公共地图抓取。

### 3.8 增加“质量检查 Checklist”

建议每次生成后自动检查：

| 检查项 | 目的 |
|--------|------|
| 道路是否断裂 | 避免交通系统失败 |
| 路口是否重叠 | 避免车辆寻路异常 |
| 建筑是否压到道路 | 避免穿插 |
| 建筑高度是否异常 | 避免 OSM 脏数据导致摩天楼 |
| Tile 边界是否有缝 | 避免 World Partition 加载时露缝 |
| 坐标是否偏移 | 避免 Cesium / UE / Houdini 对不齐 |
| 实例数量是否超预算 | 避免 Draw Call / 内存爆炸 |
| 材质槽数量是否超标 | 避免 Nanite 场景性能下降 |

## 4. 对各文档的具体优化建议

### 4.1 `01_地图数据获取与API对比.md`

建议补充：

- QGIS / GDAL 作为正式预处理工具。
- GeoJSON / SHP / CityGML / 3D Tiles / IFC / CAD 数据格式对比。
- 国内坐标系专项：WGS84、GCJ-02、BD-09、CGCS2000。
- Mapbox 仅作为瓦片服务或自定义 API，不再作为 Houdini Labs 主路线。
- Google Photorealistic 3D Tiles 的 ToS、版权标识、计费、不可离线导出限制。

### 4.2 `02_Houdini处理流程.md`

建议补充：

- OSM 进入 Houdini 前先统一投影。
- 使用 PDG / TOPs 做 Tile 并行处理。
- 输出 manifest 和版本 hash。
- 增加道路拓扑修复：snap、fuse、intersection cleanup、lane width by highway tag。
- 增加建筑规则：高度缺失补全、按 landuse / POI / road hierarchy 推断建筑类型。

### 4.3 `03_UE5整合方案.md`

建议补充：

- Houdini Engine 普通版 vs PCG 版选择建议。
- UE 5.5 / 5.6 项目应检查 Houdini Engine 对应分支，不要只写 UE 5.3 / 5.4。
- World Partition + OFPA + Data Layers + HLOD 的组合规范。
- Bake 后资产命名、目录结构、Data Layer 归属规则。
- Cesium 与 UE 原生世界坐标混用时的 Georeferencing 策略。

### 4.4 `04_完整流程速查表.md`

建议补充一个“生产级完整流程”：

```text
数据授权确认
    ↓
GIS 预处理
    ↓
Houdini Tile 生成
    ↓
自动 QA 检查
    ↓
UE5 导入 / Cook / Bake
    ↓
World Partition 分层
    ↓
性能 Profiling
    ↓
发布包验证
```

### 4.5 `05_虚拟城市主流技术方案.md`

建议补充：

- ArcGIS Maps SDK for Unreal 与 Cesium for Unreal 的差异。
- CityEngine 更适合 GIS / 规划建筑规则，不一定适合 AAA 美术城市。
- OpenUSD / Omniverse 适合多软件协同，但不一定是 UE 游戏运行时最佳格式。
- 3D Gaussian Splatting 适合视觉重建和背景参考，当前仍不适合替代游戏级可碰撞城市资产。

### 4.6 `06_路线一详细拆解_Houdini+UE5城市管线.md`

建议补充：

- City Sample 的 Rule Processor 不是通用公开生产工具，普通项目需要自研等价的点云实例化导入器，或改用 Houdini Engine / PCG / Datasmith / Python Editor Utility。
- `.pbc` / Point Cloud Alembic 是参考范式，不一定是所有项目的标准答案。
- 对中小团队，优先使用 HDA 输出 ISM / PCG Points，而不是完整复刻 City Sample 内部工具链。
- 增加“最小可行版本 MVP”：先做 200m × 200m 城市块，再扩展到 1km² / 5km²。

## 5. 推荐的 MVP 落地路线

如果当前目标是快速验证管线，建议不要一开始复刻完整 City Sample。

### Phase 1：200m × 200m 小区块

```text
OSM bbox
    ↓
QGIS / Python 坐标转换
    ↓
Houdini 生成道路、建筑盒子、人行道
    ↓
HDA 导入 UE5
    ↓
Bake 成 Static Mesh / ISM
```

目标：验证坐标、比例、导入、材质和基础性能。

### Phase 2：1km² 可游玩样区

```text
Tile 分块
    ↓
建筑规则集
    ↓
PCG 植被和街具
    ↓
World Partition
    ↓
交通 / 行人基础测试
```

目标：验证分块、流送、实例化、交互。

### Phase 3：5km² 城市级样本

```text
PDG 批处理
    ↓
多 Data Layer
    ↓
HLOD / 性能预算
    ↓
自动 QA
    ↓
发布包验证
```

目标：验证生产级大城市管线。

## 6. 最终建议

现有文档的主线不需要推翻，建议按以下优先级补强：

1. 先补 `GIS 预处理 + 坐标系统一`。
2. 再补 `Houdini Tile / PDG / 增量 Cook`。
3. 再补 `UE5 World Partition + HLOD + PCG 版 Houdini Engine`。
4. 最后补 `国内地图数据合规和 GCJ-02 / CGCS2000 转换`。

如果项目目标是游戏 / 影视 / XR，最佳路线仍是：

```text
OSM / GIS 参考数据 + Houdini 程序化重建 + UE5 Nanite / Lumen / PCG / World Partition
```

如果项目目标是数字孪生 / 城市规划，则最佳路线应调整为：

```text
权威 GIS / 3D Tiles / Cesium / ArcGIS SDK 为主，Houdini 只做局部程序化增强
```
