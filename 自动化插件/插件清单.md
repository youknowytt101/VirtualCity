# 自动化插件 / 工具清单

> 真实项目精简版。  
> 结论：MVP 阶段不需要先开发一堆插件，甚至可以 **0 个自研插件** 跑通。  
> 本目录先作为“自动化预留位”和“后续痛点记录区”，不要一开始就投入工具开发。

---

## 1. 真实项目中的必要性判断

| 阶段 | 是否需要自研插件 | 更现实的做法 |
|---|---:|---|
| 调研验证 | 否 | 手动从 Overpass Turbo 导出 `.osm`，Houdini 手动导入 |
| MVP 小区块 | 基本不需要 | 用 SideFX Labs + Houdini Engine 跑通白盒城市 |
| 重复生产多个区域 | 可能需要 1～2 个脚本 | 自动记录 bbox、路径、参数和导出版本 |
| 大地图 / 多人协作 | 才需要工具化 | 批处理、分块、资产整理、Data Layer 自动化 |
| 正式量产 | 需要管线工具 | PDG、批量 Cook、UE 资产整理、质量检查 |

---

## 2. MVP 阶段真正需要的东西

MVP 阶段优先目标是验证流程，不是建设工具平台。

### 必须有

| 名称 | 形式 | 作用 |
|---|---|---|
| 区域记录文件 | Markdown / JSON | 记录 bbox、城市名、数据来源、下载时间 |
| Houdini 参数记录 | Markdown / JSON | 记录建筑高度系数、道路宽度、地形分辨率 |
| 手动导入流程说明 | Markdown | 确保 OSM → Houdini → UE5 的步骤可复现 |

### 可以没有

| 工具 | 暂时不用的原因 |
|---|---|
| OSM 自动下载器 | Overpass Turbo 手动导出更快，且便于检查数据 |
| DEM 自动裁切工具 | MVP 可以先不用 DEM，或手动裁切一次 |
| Hython 批量 Cook | 小区块手动 Cook 足够 |
| UE5 资产整理脚本 | 第一版资产数量少，手动整理更省时间 |
| Data Layer 自动分配 | MVP 阶段分层对象少，手动分即可 |
| Nanite 批量启用 | 第一版建筑体量简单，不是主要瓶颈 |

## 4. 第一阶段只建议做 1 个轻量配置文件

如果一定要做自动化相关内容，第一阶段只建议做一个区域配置文件，例如：

```json
{
  "area_name": "mvp_area",
  "bbox": {
    "west": 0,
    "south": 0,
    "east": 0,
    "north": 0
  },
  "source": {
    "osm": "Overpass Turbo",
    "dem": "manual_or_none"
  },
  "houdini": {
    "building_height_multiplier": 1.0,
    "road_width_multiplier": 1.0
  }
}
```

这不是插件，只是项目记录文件。

## 5. 什么时候才真的需要开发工具

只有出现以下情况，才值得开发对应工具：

| 触发条件 | 再开发什么 |
|---|---|
| 每周要下载很多个 bbox | OSM 下载脚本 |
| OSM、DEM、参考图经常错位 | 坐标检查脚本 |
| Houdini 每次要导出多种格式 | 批量导出脚本 |
| 同一套 HDA 要批量跑很多区域 | Hython / PDG 批处理 |
| UE5 导入后资产数量上百 | 资产整理脚本 |
| 大地图超过 1km² | World Partition / 分块检查工具 |

## 6. 已实现脚本

| 脚本 | 触发方式 | 功能 |
|---|---|---|
| `download_building_heights.py` | `run_download_heights.bat` 双击 | 从 Google Open Buildings 2.5D 下载建筑高度 GeoJSON，无需 Drive |

**依赖**：`uv`（已安装）+ 阿里云 PyPI 镜像 + Google Earth Engine 账号（`pty-zone` 项目）  
**首次运行**：自动打开浏览器做 OAuth 授权，凭据本地缓存，后续无需再登录。
