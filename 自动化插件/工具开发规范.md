# 工具开发规范

## 1. 总原则

- 不把官方插件本体复制到本目录。
- MVP 阶段默认不开发自研脚本。
- 只有当某一步重复出现 3 次以上，或手动操作开始影响交付效率时，才考虑脚本化。
- 所有自研脚本需要能独立说明输入、输出和依赖。
- 文件路径优先使用相对 `D:\VirtualCity` 的项目路径。
- 关键配置使用 JSON 或 YAML，避免硬编码在脚本中。
- 第一阶段以可验证为主，不追求完整平台化。

---

## 2. 命名规范

### 工具 ID

使用英文小写加下划线：

```text
coordinate_alignment_checker
houdini_batch_exporter
ue_asset_organizer
```

### 输出文件

建议带上区域名、数据类型和版本：

```text
mvp_area_osm_v001.osm
mvp_area_dem_v001.tif
mvp_city_block_v001.fbx
mvp_city_block_params_v001.json
```

---

## 3. 配置文件建议

第一阶段优先做区域配置文件，而不是自动化工具：

```json
{
  "project_root": "D:/VirtualCity",
  "area_name": "mvp_area",
  "bbox": {
    "west": 0,
    "south": 0,
    "east": 0,
    "north": 0
  },
  "coordinate_system": "WGS84",
  "output_version": "v001"
}
```

---

## 4. 日志规范

脚本运行时至少输出：

- 输入文件路径。
- 输出文件路径。
- 坐标系。
- bbox。
- 要素数量统计。
- 错误和警告。
- 运行耗时。

---

## 5. MVP 优先级

优先完成能让小范围城市跑通的人工流程：

1. 手动下载 OSM。
2. 手动导入 Houdini。
3. 手动调参并导出。
4. 手动导入 UE5 并 Bake。
5. 记录 bbox、参数、版本和问题。

暂缓：

- OSM 自动下载。
- DEM 自动裁切。
- Houdini 批量 Cook。
- UE5 资产自动整理。
- 大规模 PDG 自动调度。
- World Partition 自动分块。
- Mass AI 自动配置。
- 完整程序化建筑立面规则平台。
