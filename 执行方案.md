# VirtualCity 可执行方案

> 基于 `D:\VirtualCity` 现有调研文档整理。  
> 目标：把“地图数据 → Houdini 程序化处理 → UE5 场景整合”的调研结论转成可执行、可验收、可迭代的落地计划。

---

## 1. 当前文档结论

### 1.1 项目当前状态

`VirtualCity` 目前是一个调研型工程骨架：

```text
VirtualCity/
├── README.md
├── 原始数据/          # 当前为空，计划放 OSM / DEM
├── Houdini/           # 当前为空，计划放 .hip / .hda
├── UE5/               # 当前为空，计划放 UE5 项目
└── 调研文档/          # 已完成主要技术路线调研
```

已形成的关键调研结论：

- **Mapbox 路线不再作为主线**：SideFX Labs 内置 Mapbox 节点已移除。
- **主路线应采用 OSM + DEM + Houdini + UE5**：最适合可编辑、可控的游戏/影视/XR 虚拟城市。
- **Cesium 更适合作为参考底图或快速预览**：适合数字孪生和真实地理可视化，但流式数据不可作为离线资产自由导出。
- **UE5 侧推荐 Houdini Engine for Unreal**：支持 HDA 实时调参、Landscape、Static Mesh、ISM、Spline、World Partition。
- **完整 AAA 城市管线不可一步到位**：City Sample 级别需要大量模块资产、规则系统和 PDG 处理，应先做小范围 MVP。

---

## 2. 推荐落地路线

### 2.1 主路线

采用以下路线作为项目主线：

```text
OpenStreetMap .osm + DEM GeoTIFF
        ↓
Houdini SideFX Labs
        ↓
OSM Import / Roads / Buildings / Heightfield
        ↓
封装 HDA 或静态导出
        ↓
UE5 Houdini Engine
        ↓
Bake 为 Static Mesh / Landscape
        ↓
Nanite + Lumen + PCG 扩展
```

### 2.2 不建议作为第一阶段主线的路线

- **不建议先做完整 City Sample 级管线**：模块库、Rule Processor、Shape Grammar、Mass AI 成本太高。
- **不建议先依赖 Google Photorealistic 3D Tiles**：需要 API Key、可能产生费用，并且受 ToS 限制。
- **不建议先做全国/整城级数据**：坐标、性能、分块、Cook 时间都会放大问题。

### 2.3 第一阶段目标

先完成一个 **200m × 200m 到 500m × 500m 的可验证城市区块**：

- 能从 OSM 导入真实道路和建筑轮廓。
- 能在 Houdini 中生成道路、建筑体量、基础地形。
- 能导入 UE5，并在 UE5 中看到可运行场景。
- 能 Bake 成普通 UE5 资产，脱离 Houdini Engine 也可查看。

## 3. 阶段执行计划

## Phase 0｜环境准备

### 目标

搭建能跑通 OSM → Houdini → UE5 的基础环境。

### 推荐版本

| 工具 | 推荐版本 | 用途 |
|---|---:|---|
| Houdini | 20.x / 20.5 | 程序化生成、Heightfield、HDA |
| SideFX Labs | 跟随 Houdini 安装 | OSM Import、Roads、Buildings |
| Unreal Engine | 5.3 / 5.4 | Houdini Engine 兼容性较稳 |
| Houdini Engine for Unreal | 对应 UE 分支 | HDA 导入 UE5 |
| Cesium for Unreal | 可选 | 真实底图参考、快速预览 |

### 操作清单

- [ ] 安装 Houdini。
- [ ] 在 Houdini Package Manager 安装 SideFX Labs。
- [ ] 安装 UE5.3 或 UE5.4。
- [ ] 安装与 UE 版本匹配的 Houdini Engine for Unreal。
- [ ] 创建 UE5 项目到 `D:\VirtualCity\UE5\VirtualCityUE`。
- [ ] 新建 Houdini 工作文件到 `D:\VirtualCity\Houdini\VirtualCity_OSM_MVP.hip`。

### 验收标准

- Houdini 中能看到 SideFX Labs 节点。
- UE5 中能启用 Houdini Engine 插件。
- UE5 能成功导入一个测试 `.hda`。

## Phase 1｜数据获取与范围确定

### 目标

准备一块小范围、可反复测试的真实城市数据。

### 推荐范围

第一版不要超过：

```text
200m × 200m ～ 500m × 500m
```

优先选择：

- 道路结构清晰。
- 建筑轮廓较完整。
- 不含复杂山地或超大水体。
- 如果是中国城市，优先选 OSM 数据质量较好的区域，如上海、深圳、北京核心区。

### 数据来源

| 数据 | 来源 | 输出位置 |
|---|---|---|
| OSM 建筑/道路/绿地/水系 | https://overpass-turbo.eu/ | `D:\VirtualCity\原始数据\OSM\mvp_area.osm` |
| DEM 高程 | Copernicus / SRTM / USGS | `D:\VirtualCity\原始数据\DEM\mvp_area.tif` |
| 参考底图 | Cesium / 卫星图 / 截图 | `D:\VirtualCity\原始数据\Reference\` |

### 操作清单

- [ ] 在 Overpass Turbo 框选小范围区域。
- [ ] 导出 `.osm` 文件。
- [ ] 下载同区域 DEM，优先 GeoTIFF。
- [ ] 记录 bbox 经纬度范围。
- [ ] 保存一张参考截图，方便后续比对道路方向和建筑密度。

### 验收标准

- `原始数据/OSM/` 下存在可导入的 `.osm`。
- `原始数据/DEM/` 下存在同范围或覆盖该范围的高程文件。
- 文档中记录 bbox、城市名称、数据来源、下载时间。

## Phase 2｜Houdini 最小城市生成

### 目标

在 Houdini 中生成第一版可看的城市区块。

### 节点流程

```text
Labs OSM Import
    ├── building → Labs OSM Filter → PolyExtrude / Labs OSM Buildings
    ├── highway  → Labs OSM Roads → 道路多边形
    ├── landuse  → Labs OSM Landuse → 绿地/水体/地块
    └── water    → 独立水面几何

Height Field File
    ↓
Height Field Resize / Blur
    ↓
Height Field Project 道路压平
    ↓
Height Field Output
```

### 建筑生成策略

第一版只做体量，不做复杂立面模块：

- 如果有 `building:height`，直接使用高度。
- 如果有 `building:levels`，按 `levels × 3m` 估算。
- 如果都没有，按建筑占地面积和道路等级生成默认高度。
- 使用 3～5 种基础材质区分住宅、商业、工业、未知类型。

### 建议暴露参数

| 参数 | 用途 |
|---|---|
| `building_height_multiplier` | 全局高度系数 |
| `default_floor_height` | 默认层高 |
| `default_levels_min/max` | 缺省楼层范围 |
| `road_width_multiplier` | 道路宽度系数 |
| `terrain_resolution` | Heightfield 分辨率 |
| `preview_lod` | 预览精度 |

### 输出内容

| 输出 | 建议文件 |
|---|---|
| Houdini 工作文件 | `D:\VirtualCity\Houdini\VirtualCity_OSM_MVP.hip` |
| HDA | `D:\VirtualCity\Houdini\HDA\VC_OSM_CityBlock.hda` |
| 静态备份 | `D:\VirtualCity\Houdini\Export\mvp_city_block.fbx` |
| 高度图备份 | `D:\VirtualCity\Houdini\Export\mvp_landscape_height.png` |

### 验收标准

- Houdini 视口中能看到道路、建筑体量、地形。
- 道路与建筑没有明显错位。
- 建筑高度规则可通过参数调整。
- 能导出 HDA 或 FBX 备份。

## Phase 3｜UE5 集成与 Bake

### 目标

把 Houdini 生成结果导入 UE5，并变成可运行、可保存、可烘焙的关卡资产。

### 推荐流程

```text
UE5 创建项目
    ↓
启用 Houdini Engine 插件
    ↓
导入 VC_OSM_CityBlock.hda
    ↓
拖入 Level
    ↓
调整参数并 Recook
    ↓
Bake 为 Static Mesh / Landscape
    ↓
启用 Nanite / Lumen
```

### UE5 项目设置

- [ ] 开启 Lumen。
- [ ] 开启 Virtual Shadow Maps。
- [ ] Static Mesh 资产启用 Nanite。
- [ ] 大于 1km² 后再考虑 World Partition。
- [ ] 第一阶段暂不引入 Mass AI。
- [ ] 第一阶段 PCG 只做简单树木/路灯/垃圾桶散布。

### Data Layers 建议

| Layer | 内容 |
|---|---|
| `DL_Terrain` | Landscape / 地形 |
| `DL_Roads` | 道路、路缘石、人行道 |
| `DL_Buildings` | 建筑体量 |
| `DL_Foliage` | PCG 植被 |
| `DL_Props` | 路灯、长椅、垃圾桶等 |

### 验收标准

- UE5 中能看到完整城市区块。
- HDA 参数调整后可以 Recook。
- Bake 后不依赖 Houdini 也能打开场景。
- 场景能 Play，无明显性能或加载错误。

## Phase 4｜质量提升与可视化

### 目标

从“能跑通”提升到“能展示”。

### 优先级顺序

1. **道路正确性**：道路宽度、路口、方向比建筑细节更重要。
2. **建筑密度**：先保证城市轮廓可信，再做立面细节。
3. **材质分层**：用少量材质实例快速区分城市区域。
4. **街道家具**：用 PCG 或点位散布补路灯、树、垃圾桶。
5. **灯光氛围**：通过 Lumen、天空光、雾效增强展示效果。

### 可选增强

- 使用 Cesium for Unreal 加载真实地理底图作为参考。
- 使用 PCG Graph 沿道路散布行道树。
- 用简单规则生成窗户贴图或窗格几何。
- 对建筑按高度或用地类型设置不同材质实例。

### 验收标准

- 有 3～5 个可截图展示的视角。
- 白天和夜晚各有一套基础光照。
- 路网、建筑、地形在视觉上对齐。
- 形成一个可演示的 UE5 关卡。

## Phase 5｜从 MVP 扩展到正式管线

### 扩展方向

| 方向 | 触发条件 | 做法 |
|---|---|---|
| 扩大地图范围 | MVP 稳定后 | 从 500m 扩到 1km²，再考虑分块 |
| 模块化建筑 | 需要更高美术质量 | 建立窗户、底商、屋顶模块库 |
| Shape Grammar | 建筑风格要多样化 | 在 Houdini 中建立规则分割系统 |
| PDG 并行 | Cook 时间超过数分钟 | 按街区/地块拆分并行处理 |
| World Partition | 场景超过 1km² | UE5 中启用大世界分区 |
| Mass AI | 需要交通和人群 | Houdini 输出 traffic zone / pedestrian density |

### 正式管线目标形态

```text
OSM / 手绘规划 / DEM
        ↓
Houdini 城市生成 HDA
        ↓
建筑体量 / 道路 / 地块 / 点位属性
        ↓
UE5 Houdini Engine 或静态导入
        ↓
Nanite 模块实例化
        ↓
PCG 植被道具
        ↓
World Partition 管理大场景
```

## 4. 一周执行排期建议

### Day 1｜环境与数据

- 安装 Houdini、SideFX Labs、UE5、Houdini Engine。
- 选择小范围 bbox。
- 下载 `.osm` 和 DEM。
- 建立数据记录表。

### Day 2｜Houdini OSM 导入

- 跑通 `Labs OSM Import`。
- 分离道路、建筑、水系、绿地。
- 生成第一版道路和建筑体量。

### Day 3｜地形与对齐

- 导入 DEM 为 Heightfield。
- 将 OSM 数据与地形对齐。
- 处理道路压入地形。
- 修正比例和单位。

### Day 4｜HDA 封装

- 暴露关键参数。
- 输出 `OUT_geo`、`OUT_landscape`。
- 保存 `VC_OSM_CityBlock.hda`。
- 做小范围参数测试。

### Day 5｜UE5 导入

- 创建 UE5 项目。
- 导入 HDA。
- 调整参数并 Recook。
- Bake 为 Static Mesh / Landscape。

### Day 6｜展示效果

- 开启 Nanite / Lumen。
- 增加基础材质。
- 增加 PCG 植被或路灯散布。
- 设置 3～5 个镜头点。

### Day 7｜复盘与扩展计划

- 记录问题清单。
- 固化目录和命名规范。
- 决定是否扩大范围到 1km²。
- 决定是否进入模块化建筑阶段。

## 5. 关键风险与处理策略

| 风险 | 表现 | 处理策略 |
|---|---|---|
| OSM 建筑无高度 | 建筑全是同高或缺失高度 | 用 `building:levels`、面积、随机规则补全 |
| 坐标错位 | 道路、地形、建筑不重合 | 统一投影，优先转换到 UTM 米制坐标 |
| 国内坐标偏移 | WGS-84 与 GCJ-02 不一致 | 明确数据源坐标系，必要时做坐标转换 |
| HDA Cook 慢 | UE5 调参等待过长 | 使用 200m 小范围代理调参，正式范围最后生成 |
| UE 单位错误 | 导入后比例大 100 倍或小 100 倍 | Houdini 以 1 unit = 1m 建模，UE 导入统一 ×100 |
| Cesium 版权/费用 | API Key、在线流式、版权显示 | Cesium 只作为参考或在线展示，不作为离线资产来源 |
| 美术质量不足 | 建筑像白盒盒子 | MVP 后再进入模块化立面和材质实例阶段 |

## 7. 推荐下一步

按成本和确定性排序，下一步建议直接做以下三件事：

1. **确定测试区域 bbox**：选一个 200m～500m 的小范围。
2. **下载 OSM 数据**：先不追求 DEM，优先跑通道路和建筑。
3. **在 Houdini 中建立 MVP 节点网络**：先生成白盒城市，再考虑 UE5 美术效果。

完成这三步后，项目就能从“调研文档”进入“可运行原型”。
