# 工具开发规范

## 1. 总原则

- 不把官方插件本体复制到本目录。
- MVP 阶段默认不开发自研脚本。
- 只有当某一步重复出现 3 次以上，或手动操作开始影响交付效率时，才考虑脚本化。
- 所有自研脚本需要能独立说明输入、输出和依赖。
- 文件路径优先使用相对 `D:\VirtualCity` 的项目路径。
- 关键配置使用 JSON 或 YAML，避免硬编码在脚本中。
- 第一阶段以可验证为主，不追求完整平台化。

## 3. 配置文件建议

第一阶段优先做区域配置文件，而不是自动化工具：

```json
{
  "project_root": "<repo-root>",
  "area_name": "mvp_area",
  "bbox": {
    "west": 0,
    "south": 0,
    "east": 0,
    "north": 0

  "output_version": "v001"
```

## 5. MVP 优先级

优先完成能让小范围城市跑通的人工流程：

1. 手动下载 OSM。
2. 手动导入 Houdini。
3. 手动调参并导出。
4. 手动导入 UE5 并 Bake。
5. 记录 bbox、参数、版本和问题。

暂缓：

- OSM 自动下载。
- DEM 自动裁切。
- Houdini 批量 Cook。
- UE5 资产自动整理。
- 大规模 PDG 自动调度。
- World Partition 自动分块。
- Mass AI 自动配置。
- 完整程序化建筑立面规则平台。

## 6. 已知 Bug 清单与修复规则

> 格式：`[ID] 类别 | 症状 | 根因 | 修复`

### D 系列：数据文件问题

**D-001** 建筑 GeoJSON geometry=null
- **症状**：Houdini 视口无建筑，`extract_buildings` pts=0
- **根因**：`pattaya_sai6_mvp_v2_buildings_height_v001.geojson` 是从 GEE 导出的高度点云（无多边形坐标），geometry 全部为 null
- **修复**：`active_area.json` 的 `buildings_file` 必须使用含 Polygon geometry 的文件（如 `pattaya_sai6_buildings_height_v001.geojson`）
- **预防**：`validate_data.py` 已加入 D-001 检查，运行前必须通过

**D-002** 建筑高度全为默认值 10m
- **症状**：所有建筑高度一致，城市轮廓单调
- **根因**：`pattaya_sai6_mvp_buildings_v001.geojson` 有多边形坐标但缺少 `height` 属性
- **修复**：必须使用同时包含 geometry + height 的文件（文件名含 `_height_` 的版本）
- **预防**：`validate_data.py` D-002 检查：若 avg_height==10.0 且 min==max 则报错

**D-003** 海岸线区域 DEM 出现"水片"
- **症状**：视口出现大面积 Y=0 平面（看似水面）
- **根因**：bbox 包含海洋区域，NASADEM 对海洋赋值 0m，DEM 网格在海洋部分全平
- **修复**：沿海区域需添加陆地掩膜（land mask）或将 bbox 限制在纯陆地范围内
- **预防**：下载 DEM 前检查 bbox 是否跨越海岸线

### H 系列：Houdini 问题

**H-001** Houdini 重启后 hip 未自动加载
- **症状**：RPYC 连接成功但场景为空（`hou.hipFile.path()` 返回 `untitled`）
- **修复**：所有脚本在连接后检查 `'untitled' in hou.hipFile.path()`，若是则自动 load
- **预防**：`export_and_import.py` 的 `connect_hou()` 函数已内置此检查

**H-002** 临时裁剪节点重启后不可靠
- **症状**：`_bld_clip`/`_road_clip` 等临时节点在 hip 重新保存后偶发丢失或连接断开
- **修复**：裁剪节点永久化为 `bld_clipped` / `road_clipped`，不使用 `_` 前缀临时命名
- **预防**：`_make_permanent_clips.py` 已执行，节点已固化在 hip 文件中

**H-003** Python SOP 大批量删除 prim 导致崩溃
- **症状**：调用 `geo.deletePrims(to_del)` 删除数万 prims 时 Houdini 崩溃
- **修复**：改用 VEX attribwrangle 标记 `i@del` + blast SOP 删除，不在 Python 层迭代删除
- **预防**：禁止在 Python SOP 中循环调用 `deletePrims`

**H-004** VEX foreach 中调用 `removeprim()` 导致崩溃
- **症状**：在遍历 prim 的 wrangle 中直接 `removeprim(0, @primnum, 1)` 触发越界崩溃
- **修复**：统一改为双步：wrangle 标记 `@del` → blast SOP 清除
- **预防**：VEX 中永远不在遍历自身 prim 时 `removeprim`

### E 系列：导出问题

**E-001** 单次 RPYC 连接多个 FBX 导出时崩溃传染
- **症状**：第一个 FBX 导出成功，第二个崩溃后整个脚本终止
- **修复**：每个 FBX 使用独立的 RPYC 连接（`export_one()` 函数内 connect+close）
- **预防**：`export_and_import.py` 已改为逐个独立连接导出

**E-002** 导出空几何导致 FBX 文件损坏
- **症状**：导出成功但 FBX 为 0 bytes 或 UE5 导入失败
- **修复**：导出前检查 `geometry().intrinsicValue('pointcount') > 0`
- **预防**：`export_one()` 已加入空几何跳过逻辑

## 8. 追加 Bug（2026-05-22）

**H-005** `dem_terrain` Python SOP 非正方形格网缺面片
- **症状**：地形出现"补丁"空洞，`prims` 比理论值少（如 4356 vs 4488）
- **根因**：`n = int(math.sqrt(len(rows)))` 对 67×69 非正方形网格取整为 67，最后 2 行点未连面
- **修复**：改为 `xs=set(unique x); ncols=len(xs); nrows=len(rows)//ncols`，再用 `ncols` 做内循环
- **预防**：DEM Python SOP 禁用 `sqrt` 推断格网尺寸，必须从数据实际坐标推断

**H-007** 道路地形吸附失效（Ray SOP + attribwrangle class 双重错误）
- **症状**：65-100% 道路点停在 Y=0，视口中道路全部悬空或沉底
- **根因 A**：`snap_roads_to_terrain1` 为 Ray SOP，`direction=(0,0,0)` 全零 + `dirattrib=N`，道路无 N 属性 → ray 从不发射
- **根因 B**：替换为 attribwrangle 时误用 `class=0`（Detail，只跑1次），应为 `class=2`（Point，每点执行）
- **修复**：用 attribwrangle + `xyzdist` VEX + `class=2(Point)` 替代 Ray SOP
- **VEX**：`xyzdist(1,@P,hit_prim,uvw); @P.y = primuv(1,"P",hit_prim,uvw).y;`
- **预防**：attribwrangle 替换 Ray SOP 时必须显式设 `class=2`；点级别吸附禁用 Ray SOP

**H-006** `dem_import` Python SOP 硬编码 CSV 路径
- **症状**：换区后 Houdini DEM 仍读旧区域数据，视觉正确但 RPYC 检查 pts 不对
- **修复**：改为 `with open(active_area.json) as f: CSV_FILE = json.load(f)['dem_csv']`
- **预防**：所有 Python SOP 禁止硬编码文件路径，必须从 `active_area.json` 读取

**H-008** 道路条带侧边顶点埋入地形（`road_strips` 扩宽后未二次吸附）
- **症状**：视口中坡面道路左右边缘陷入地形，Y 值低于相同 XZ 位置的地形高度
- **根因**：`road_strips` Python SOP 生成侧边点时只继承中心线 Y 值，坡面上侧边 XZ 处的实际地形高度不同
- **修复**：在 `road_strips` 后新增 `snap_road_strips` attribwrangle（class=2），对所有条带顶点做二次 `xyzdist` 投影，加 +0.15m 防 z-fighting
- **VEX**：`xyzdist(1,@P,hit_prim,uvw); @P.y = primuv(1,"P",hit_prim,uvw).y + 0.15;`
- **预防**：任何从中心线扩宽生成的网格（road_strips、polywire 替代等）必须在最终输出前做一次全顶点地形投影

**H-009** 地形网格过稀（30m 数据源 → 4488 面/2km²）
- **症状**：Houdini/UE5 中地形轮廓呈明显阶梯/棱角，坡度过渡不自然，布线明显可见
- **根因**：NASADEM/SRTM 30m 精度，每格 30m×30m = 900m²，远粗于建筑（~10m）和道路（~7m）
- **修复**：在 `dem_terrain` 后新增 `dem_subdivide` SOP，`algorithm=4`（OpenSubdiv Bilinear，线性插值不改高程），`iterations=2`（×16 面片，等效 ~7.5m 分辨率）
- **注意**：`iterations=2` 是两轮细分 = ×4×4 = ×16，非 ×2；Bilinear 仅线性插值，不产生 Catmull-Clark 平滑变形
- **预防**：DEM 导入后必须接 Subdivide Bilinear 加密；吸附节点（snap_bld_to_terrain、snap_roads_to_terrain1）的 input1 须指向 `dem_subdivide` 而非原始 `dem_terrain`

## 数据源规范（Session 11 更新）

### 建筑数据：Overture Maps + Google Open Buildings 混合模式

- **脚本**：`download_overture_buildings.py`
- **策略**：
  1. 先从 Overture Maps 下载建筑轮廓（Overture 精度优于 OSM，融合多源）
  2. 检查高度覆盖率，若 < 80% 自动调用 Google Open Buildings（GEE）补充高度
  3. 用 shapely STRtree 空间 join，50m 范围内最近建筑取 Google 高度
- **泰国实测**：Overture 原生高度覆盖率约 2%，Google join 后达 98%
- **高度优先级**：Overture height > num_floors×3m > Google 空间 join > 默认 8m
- **OSM 建筑**：已从 Overpass 查询中移除（`way["building"]` 不再下载），建筑全部来自 Overture

### 本地 Tile 缓存系统

- **目的**：消除每次换区重复联网，OSM/DEM/建筑步骤从 ~100s 降到 < 3s
- **预缓存**：`uv run python cache_city_data.py --city pattaya`（城市级 bbox 一次性下载）
- **自动命中**：`set_area.py` 步骤 2/3/4 优先调用 `_tile_cache.py` 检查覆盖，命中则本地裁剪
- **缓存文件**：`RawData/_tiles/` — `_index.json` + `{city}_dem.tif` + `{city}_osm.osm` + `{city}_bld.geojson`
- **新城市流程**：先运行一次 `cache_city_data.py --city xxx --bbox ...`，约 5 分钟；后续所有小区域秒级完成
- **注意**：缓存 bbox 必须完整包含目标小区域，否则自动回退网络下载

### 几何顶点色系统（三类独立控制）

- **配置位置**：`_recook_new_area.py` 顶部 `COLORS` 字典，改完重跑脚本即生效
- **默认配色**：道路纯白 (1,1,1) / 建筑中灰 (0.55,0.55,0.55) / 地形深灰 (0.25,0.25,0.25)
- **节点结构**：`bld_clipped→bld_color` / `road_clipped→road_color` / `dem_subdivide→terrain_color` → `merge_all`
- **技术原因**：Houdini Merge SOP 合并带 Cd 和不带 Cd 的几何时，无 Cd 的一方会被默认补为白色 (1,1,1)；三类必须全部显式赋色才能正确区分
- **FBX 导出**：`Cd` 顶点色随 FBX 导出，UE5 中需在材质中接入 Vertex Color 节点才能显示

### 道路分级宽度系统（Session 12 更新）

- **节点**：`road_width` attribwrangle，class=Primitive，写入 `f@half_width`
- **VEX 管理**：`ROAD_WIDTH_VEX` 常量在 `_recook_new_area.py` 中，每次 recook 自动写入节点
- **14 级宽度映射**（half_width，单位 m）：

| 类型 | 半宽 | 全宽 |
|------|------|------|
| motorway | 5.0 | 10m |
| trunk | 4.5 | 9m |
| primary | 4.0 | 8m |
| motorway/trunk_link | 3.5/3.0 | 7/6m |
| secondary | 3.5 | 7m |
| tertiary / pedestrian | 2.5 | 5m |
| residential / unclassified | 2.0 | 4m |
| service / track | 1.5 | 3m |
| footway / path / cycleway | 0.75 | 1.5m |
| steps | 0.6 | 1.2m |
| 其他（默认） | 1.5 | 3m |

### 道路贴地修复（Session 12 更新）

- **问题**：DEM 近海平面区域 Y 值略为负数，道路吸附后 Y_min 出现负值（如 -0.59m）
- **修复**：`ROAD_DRAPE_VEX` 中改为 `@P.y = max(tp.y, 0.0) + 0.15`，确保道路不低于海平面
- **效果**：`snap_road_strips` Y_min 从 -0.59m 升为 ≥ 0.15m
